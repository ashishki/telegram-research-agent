## Block 1 — How .cursorrules Actually Works and Why It Matters for Agent Projects

Time: 45 minutes
Level: accessible now

The Cursor post is about something real: when you define behavioral rules for an LLM-assisted tool at the project level, output quality and consistency measurably improve. This is the same principle behind system prompt engineering in agents. Understanding how .cursorrules encodes context, constraints, and style is directly transferable to how you write system prompts for gdev-agent and film-school-assistant.

Read the Cursor rules documentation:
https://docs.cursor.com/context/rules-for-ai

Browse the community-curated .cursorrules repo to see real examples across different stacks:
https://github.com/PatrickJS/awesome-cursorrules

After reading, do this one mental mapping exercise: for each rule pattern you find useful (e.g. "always return typed responses", "explain reasoning before code"), ask yourself where the equivalent constraint belongs in your agent's system prompt.

What to take away: rules files are a form of instruction tuning at the project level. The patterns that work there work in system prompts. This directly improves telegram-research-agent digest quality and film-school-assistant tutoring behavior.

## Block 2 — Eval Pipeline Design for LLM Outputs

Time: 75 minutes
Level: accessible now, with one stretch section

gdev-agent has an eval pipeline listed as an open concern. Most developers skip evals until something breaks. Understanding the design space now will save you significant debugging time later.

Start here — Hamel Husain's practical post on LLM evals (one of the clearest practitioner-written explanations available):
https://hamel.dev/blog/posts/evals/

Then read the RAGAS framework README to see how evaluation is structured for retrieval-augmented systems (relevant to telegram-research-agent which does clustering and insight generation):
https://github.com/explodinggradients/ragas

Stretch section (skip if time-pressed): the original Eval-driven development framing in the OpenAI evals repo README:
https://github.com/openai/evals/blob/main/README.md

What to take away: evals are not just test assertions. They are a feedback loop that tells you whether prompt changes, model swaps, or retrieval changes actually improved output. For gdev-agent, the minimum viable eval is: define 3-5 golden examples with expected outputs, score new outputs against them after each change.
