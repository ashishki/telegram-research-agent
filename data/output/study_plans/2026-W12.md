## 🗓️ Block 1 — 60 min | RAG Fundamentals: Why and How It Actually Works

> **Goal:** Build a mental model of RAG from first principles — understand the problem it solves before touching code.

### Why this matters first
Your `telegram-research-agent` is a real retrieval problem: posts come in, you need to query them meaningfully. Without understanding RAG's architecture, you'll cargo-cult solutions. Understand the retrieval-generation split first.

### Step 1 — Read (20 min)
**"Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"** — the original RAG paper by Lewis et al. (2020). Don't read it like a paper, read the Abstract + Introduction + Figure 1 only.

🔗 https://arxiv.org/abs/2005.11401

**What to extract:**
- Why parametric memory (LLM weights) alone is insufficient
- What the retriever + generator split looks like
- Why RAG is cheaper to update than fine-tuning

### Step 2 — Watch (25 min)
**"RAG from scratch"** by LangChain — a focused code-first walkthrough of naive RAG → advanced RAG patterns.

🔗 https://www.youtube.com/watch?v=sVcwVQRHIc8

**Watch with notebook open** — pause when code appears, type it yourself.

### Step 3 — Connect to your code (15 min)
Open `ashishki/telegram-research-agent` locally. Answer these 3 questions in a comment block at the top of your main agent file:
1. Where does retrieval currently happen (or where *should* it happen)?
2. What is your vector store / index? (Or is there none yet?)
3. What is the "generation" step — which LLM call receives retrieved context?

> 💡 **WHY before HOW checkpoint:** RAG solves the problem that LLMs hallucinate on private/recent data. Your Telegram posts are exactly this: private, recent, not in any model's training data.

## 🗓️ Block 2 — 60 min | Advanced RAG Patterns: Chunking, Embedding, and Re-ranking

> **Goal:** Understand the 3 decisions that make or break a RAG system in production.

### The 3 decisions framework
Before writing code, know that RAG quality lives and dies by:
1. **Chunking strategy** — how you split documents
2. **Embedding model choice** — how you encode meaning
3. **Re-ranking** — how you sort retrieved candidates before sending to LLM

### Step 1 — Read (20 min)
**"Advanced RAG Techniques: An Illustrated Overview"** by Towards AI — best single visual explainer of chunk strategies, hybrid search, and re-ranking.

🔗 https://pub.towardsai.net/advanced-rag-techniques-an-illustrated-overview-04d193d8fec6

> If paywalled: 🔗 https://arxiv.org/abs/2312.10997 (the "RAG Survey" paper — read Section 3 only, ~6 pages)

### Step 2 — Hands-on code lab (30 min)
Work through the official LangChain RAG tutorial — specifically the **"Indexing"** and **"Retrieval"** sections. This maps directly to your Telegram agent use case.

🔗 https://python.langchain.com/docs/tutorials/rag/

**Concrete exercise while reading:**
- Swap out the example documents for 10 real Telegram post strings (copy from your dataset)
- See what the chunks and embeddings look like
- Don't build a full app — just run the retriever and print the top-3 results

### Step 3 — Design decision for your project (10 min)
Write a 5-bullet decision doc (can be in a GitHub issue or a `NOTES.md`) for `telegram-research-agent`:
- Chunk size choice and why
- Embedding model (OpenAI `text-embedding-3-small` vs local `nomic-embed-text`)
- Vector store choice (Chroma for local dev? Pinecone for prod?)
- Re-ranking: yes/no for now?
- Top-K to retrieve before generation

> 🔶 **Come back later:** Cross-encoder re-ranking (Cohere Rerank, BGE reranker) is a stretch topic — revisit after you have a baseline RAG working.

## 🗓️ Block 3 — 60 min | AI CLI Tools: Claude Code & Codex CLI in Your Real Workflow

> **Goal:** Stop using these tools as "fancy autocomplete." Learn the mental model for agentic coding assistants and wire them into your actual projects.

### Why this is not a "nice to have"
Claude Code and Codex CLI are **agentic loops** — they read your file system, write code, run tests, and iterate. Understanding their architecture makes you a better agent builder, not just a faster coder.

### Step 1 — Read the architecture (15 min)
**Claude Code: How it works** — Anthropic's official overview. Focus on the "how Claude Code approaches a task" section — it's a mini agent architecture lesson.

🔗 https://docs.anthropic.com/en/docs/claude-code/overview

Then skim OpenAI Codex CLI README for comparison — notice the similarities in the agentic loop:

🔗 https://github.com/openai/codex

**Key mental model to extract:** Both tools are: `observe (read files) → plan (CoT) → act (write/run) → observe again`. This is the **ReAct loop**. You'll build this yourself one day.

### Step 2 — Practical session on YOUR code (35 min)
Pick one of these two tasks (both are real TODO items on your projects):

**Option A — `gdev-agent`:** Open Claude Code in the repo root. Give it this prompt:
```
Read the existing agent architecture. Identify the entry point, 
the LLM call, and any tool/function definitions. Then write a 
ARCHITECTURE.md summarizing the flow in plain English.
```

**Option B — `telegram-research-agent`:** Give it:
```
Read the codebase. Identify where I could add a RAG retrieval step 
before the LLM call. Propose the minimal code change (don't implement 
yet) and explain your reasoning.
```

**While it runs:** Watch *what files it reads*, *what it skips*, *how it structures its output*. You're studying its agent behavior, not just accepting its output.

🔗 Install Claude Code if not yet: https://docs.anthropic.com/en/docs/claude-code/setup

### Step 3 — Reflect and log (10 min)
Write 3 sentences in a `dev-log.md` (or your notes app):
1. What task did the tool complete autonomously vs. where did it need your guidance?
2. One failure mode you observed (hallucinated import? wrong file path?)
3. How would you design a better "context window" for it — what files should it always read first?

> 💡 This reflection directly feeds your `OpenClaw_Evaluation_Lab` project — evaluation of agent behavior is exactly what that repo seems designed for.
